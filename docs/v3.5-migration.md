# v3.5 Migration — Batched Validity Proofs

This doc covers the v3.5 architecture, the operational concerns it
introduces (ALT rotation), and the planned phasing of the legacy
per-match validity path's retirement.

## What changed in v3.5

The v3.1 settlement flow required **two separate Groth16 proofs per
match**:

```
v3.1 — per-match (DEPRECATED):
  verify_valid_create  (Groth16 over 14 VALID_CREATE public inputs)
    → writes ValidCreateMarker PDA
  verify_valid_price   (Groth16 over 2  VALID_PRICE  public inputs)
    → writes ValidPriceMarker PDA
  tee_forced_settle    (reads both markers, asserts binding, closes both)
```

v3.5 collapses both into **one Groth16 proof attesting up to N=16
matches in a single batch**:

```
v3.5 — batched (PRODUCTION):
  verify_match_batch   (Groth16 over a single Merkle root committing
                        N=16 per-slot leaves; each leaf inlines the
                        same VALID_CREATE + VALID_PRICE constraints
                        the per-match path verified separately)
    → writes BatchValidityMarker PDA seeded by the root
  tee_forced_settle_batched
                       (recomputes the leaf from the settle payload,
                        walks a depth-4 Merkle inclusion path,
                        asserts the marker exists at the derived PDA)
```

The on-chain VALID_CREATE + VALID_PRICE constraint logic is
byte-identical between paths; only how it's bundled into proofs
differs.

### Cryptographic equivalence

- A valid `verify_match_batch` proof attests that for ALL 16 slots
  (real + dummy-padded):
  - The 6 note-commitment openings (`Poseidon7(DOMAIN_NOTE, ...)`)
    bind the public commitments to the private witness values.
  - The conservation laws hold per slot.
  - `quote_amount === base_amount * clearing_price` per slot.
  - All u64 values are range-checked.
- The Merkle inclusion proof verifier in `tee_forced_settle_batched`
  ensures that the specific payload being settled corresponds to one
  of the proven leaves.

So a successful batched settle is **strictly at least as secure as
the per-match path** for the same match.

### Throughput / cost wins

| Metric | v3.1 per-match | v3.5 batched (N=16 full batch) |
|---|---|---|
| TEE proof-gen time | ~40s per match × 16 = ~10 min | ~7s for the whole batch (M1) |
| Per-batch verify txs | 2 × 16 = 32 | 1 |
| Per-batch marker rent | 49 B × 2 × 16 = 1,568 B | 49 B × 1 = 49 B |
| Sparse-batch overhead | 0 | One extra ALT-create + 1-slot wait |

## Operational concern: Address Lookup Table rotation

The batched settle tx is ~52 bytes over the 1232-byte cap unless the
per-batch derived PDAs (note_lock_a/b/e/f + batch_validity_marker)
are looked up via an Address Lookup Table. The
`packages/sdk/tests/devnet-trade-flow.test.ts` validation creates
one fresh ALT per batch — fine for tests, wasteful in production.

### Rolling-ALT pool (recommended for matchers)

Solana's ALT primitives:
- Max 256 addresses per ALT.
- A new ALT is unusable for ~1 slot (~400 ms) after creation.
- After `deactivate`, an ALT enters a 512-slot (~3.4 min) cooldown
  before its rent is reclaimable.

A real matcher should maintain a **two-ALT rolling pool**:

```
At all times:
  current_alt   — active; accumulates up to ~246 entries
  previous_alt  — recently deactivated; serving its 512-slot cooldown

Per batch:
  1. Derive the 5 per-batch PDAs.
  2. extend current_alt with those 5 addresses (one tx).
  3. Use current_alt + the static settle ALT in the settle tx.

When current_alt's entry count > 246:
  1. Mark current_alt for deactivation.
  2. Promote previous_alt's freshly-cooled replacement to current_alt.
  3. (Optionally) reclaim rent from the just-cooled ALT.
```

At 4 batches/minute, an ALT fills in ~12 minutes — well under the
24-minute buffer two ALTs in a pool provide. **No vault- or SDK-side
code changes are required**; this lives entirely in the matcher's
operational layer.

### Anti-patterns to avoid

- **Creating a fresh ALT per match** — costs an extra tx per match
  and burns rent. Only acceptable for one-shot validation tests.
- **Sharing a single ALT across the matcher and the validity proof
  ix** — works but couples lifecycles and complicates deactivation.
- **Letting deactivation backlog grow** — accumulated rent in
  deactivated-but-uncooled ALTs adds up. Reclaim rent on a schedule.

## Phase 1c plan — retirement of the per-match ixs

The per-match `verify_valid_create` + `verify_valid_price` ixs +
their `ValidCreateMarker` / `ValidPriceMarker` state structs are
**cryptographically obsoleted** by `verify_match_batch`. They remain
on-chain for two non-cryptographic reasons:

1. The batched path is one devnet run old. A 2-week confidence
   window is the minimum to surface any edge cases (sparse-batch
   latency, ALT rotation hiccups, sequencer behavior under load,
   etc.).
2. Hard deletion would break the existing change-note + ER tests
   until they're migrated to the batched path.

## What's NOT being deprecated

To avoid confusion: the following remain in v3.5 unchanged.

- `circuits/valid_wallet_create/` — used by `create_wallet`, not
  settlement.
- `circuits/valid_input/` — used by `lock_note` (each lock is still
  per-note, can't be batched across orders).
- `circuits/valid_spend/` — used by `withdraw` (each withdrawal is
  user-initiated, not batchable).
- `tee_forced_settle`'s **canonical_payload_hash** + Ed25519 sig
  verification logic. Even after Phase 1c-hard, the batched settle
  uses these.
